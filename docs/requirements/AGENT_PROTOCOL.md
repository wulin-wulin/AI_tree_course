# Agent Collaboration Protocol

本文档说明本项目中“需求讨论窗口”和“执行智能体”的协作协议。

## 协作角色

### 需求讨论窗口

由人类用户和需求协作智能体共同使用，负责：

- 讨论产品方向、用户感受和设计需求。
- 编写与维护轮次需求文档。
- 维护 `docs/requirements/current_state.md`。
- 审核执行智能体的实现结果。
- 审核通过后，将真实项目状态回写到 `current_state.md`。

### 执行智能体

由另一个窗口承担，负责：

- 阅读 `docs/next_upgrade_prompt.md`。
- 阅读协议、当前状态和当前轮次需求。
- 只实现当前轮次中状态为 `Ready` 的需求。
- 执行完成后，把反馈写回当前轮次文档的“执行反馈”区域。
- 不直接维护 `docs/requirements/current_state.md`。

## 文件职责

### `docs/next_upgrade_prompt.md`

当前执行入口文件。

它只负责告诉执行智能体：

- 当前执行哪一轮。
- 需要阅读哪些文件。
- 反馈应该写到哪里。
- 有哪些执行边界和验证要求。

它不是正式需求归档，也不保存历史反馈。

### `docs/requirements/current_state.md`

当前项目状态说明。

它描述项目已经实现的真实状态、当前能力和已知限制。该文件由需求讨论窗口维护，执行智能体默认只读。

### `docs/requirements/rounds/Rxxx_*.md`

正式轮次需求文档。

每个轮次文档同时包含：

- 执行前需求。
- 验收标准。
- 执行智能体反馈。
- 需求讨论窗口审核记录。

执行智能体完成任务后，应把反馈写在当前轮次文档的“执行反馈”区域。

### `docs/requirements/backlog.md`

未进入正式轮次的想法池。

执行智能体不应主动实现 backlog 中的内容，除非当前轮次需求明确引用。

## 轮次状态

- `Draft`：需求讨论中，执行智能体不能实现。
- `Ready`：需求已确认，执行智能体可以实现。
- `Implemented`：执行智能体已完成实现并填写反馈。
- `Reviewed`：需求讨论窗口已完成审核。
- `Accepted`：审核通过，并已回写当前状态。

## 执行智能体工作流程

1. 阅读 `docs/next_upgrade_prompt.md`。
2. 按入口提示读取本协议、`current_state.md` 和当前轮次文档。
3. 检查当前轮次状态。
4. 如果状态不是 `Ready`，停止实现，并在回复中说明原因。
5. 如果状态是 `Ready`，按轮次文档实现需求。
6. 实现完成后运行要求的验证命令。
7. **涉及界面 / 视觉的改动，完成后必须自检视觉**：用项目的 Playwright 截图工作流（`npm run visual:review` 或 `.claude/scripts/screenshot-ui.mjs`）截取相关页面，**自己查看截图核对效果是否符合需求**；不符合或明显不好看就**微调后重新截图**，如此小步迭代，直到自己看着合格再报告完成。不要把仅"能跑通构建"当作视觉需求达标。
8. 将执行结果写入当前轮次文档的“执行反馈”区域（视觉改动应说明已自检的截图与微调情况）。
9. 在最终回复中简要说明实现概况、验证结果和反馈位置。

## 反馈要求

执行反馈必须写入当前轮次文档，不写入 `current_state.md`。

反馈至少包含：

- 实现概况
- 已完成需求
- 未完成或部分完成
- 执行中发现的问题
- 认为需求不合理或需要澄清的点
- 修改文件清单
- 验证结果

## 禁止事项

- 不要直接改写 `docs/requirements/current_state.md`，除非需求讨论窗口明确要求。
- 不要实现 `Draft` 状态的需求。
- 不要主动实现 backlog 中未被当前轮次引用的内容。
- 不要修改其他历史轮次需求，除非当前任务明确要求。
- 不要把执行反馈写到 `docs/next_upgrade_prompt.md`。
